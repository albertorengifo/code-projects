# Task Routing

## Use This Routing Guide

- route architecture or repository-shape work through planning first
- route small implementation tasks directly when the outcome is already clear
- route cross-cutting changes through docs review before expanding scope

## Examples

- `docs/architecture` for system decisions
- `apps/web` for frontend implementation tasks
- `apps/api` for backend implementation tasks
- `packages/shared` for contracts shared across both sides

## Rule Of Thumb

If a task changes more than one area, define the shared boundary first.
